# Contributing

Corrections, additions, and platform-specific notes are welcome.

## What this repository is for

This is a practitioner-authored technical series on connected device
security. Contributions should add genuine value to a working engineer
— not restate what is already in vendor documentation or standards
summaries available elsewhere.

## Before opening a pull request

**Technical claims require a cited primary source.**
Every technical claim must be supported by a normative source
(standard, specification, regulation), a peer-reviewed publication,
or a vendor's official public documentation. Secondary sources and
blog posts are not sufficient. If you cannot cite the primary source,
the claim should not be in the text.

**Platform-specific content must be clearly scoped.**
If a section describes behaviour specific to one platform, SoC family,
or OS, that scope must be stated explicitly. Do not present
platform-specific observations as universal properties.

**No vendor-confidential or NDA-covered material.**
Do not submit content derived from licensing agreements, NDAs, or
non-public documentation. This includes DRM integration guides,
vendor-restricted HAL specifications, and customer-specific designs.

**No customer or employer attribution without explicit permission.**
Case study contributions must either use publicly attributed platforms
(clearly identified as such) or be fully anonymised ("a representative
Cortex-A SoC", "a 5G-capable automotive TCU").

## What makes a good contribution

- A correction to a technical claim, with a primary source citation
- A platform-specific note that extends a general principle to a
  named platform (e.g. "On NXP iMX93, the equivalent of Level 2
  secret protection is provided by EdgeLock enclave via...")
- A new case study that illustrates an existing principle in a domain
  not yet covered (industrial OT, medical devices, space systems)
- A regulatory update when a standard cited in the series is revised
